Previously Patrick was the technical lead for the Web Connection feature for VisualAge for Smalltalk and co-developed the Smalltalk bindings for CORBA.

Session Description

T1 - Smalltalk in the Network Vehicle


Patrick Mueller

"The network vehicle" is a concept car with wireless Internet connections, voice control, heads up display, and other hardware and software innovations for automobiles. Smalltalk is used to integrate many systems, implement much of the logic on the car including voice recognition, and interface with existing networks on the car. The session will talk about how and why Smalltalk was used to develop this vehicle.

Design Patterns: Actors
  • Smalltalk application designed called for ~20 independent Smalltalk processes
  • Needed an interprocess communication mechanism
  • Used Actors
  • An Actor is
    • A process which receives messages from a queue
    • Queue is Process-safe, so many objects can write to it at once
    • Actor process is the only process to read the queue
  • Easy to implement - a BlockingQueue class which:
    • uses a mutex Semaphore to read and write 'atomically'
    • uses an event Semaphore to allow blocking read
    • OrderedCollection to hold 'messages'

Design Pattern: Application Variables
  • Smalltalk application designed called for ~10 independent subsystems
  • Needed to be started/stopped/sent messages
  • Used Application variables (see IBM Smalltalk: User's Guide: Namespace support)
  • Declaring application variables 
    {application} class >> _Pragma_{application}Globals
	"%%PRAGMA DECLARE
		(name: AppVariable1 
			isPool: false 
			isConstant: false 
			valueExpression: 'nil')
		(name: AppVariable2 
			isPool: false 
			isConstant: false 
			valueExpression: 'nil')
	"
  • Defining application variables 
    {application} class >> loaded
	AppVariable1 := AppVariable1Class new.
	AppVariable2 := AppVariable1Class new.
  • Using application variables
    • Make sure application variable is used in prereq's application variable is declared/defined in.
    • Use the variable like any other global variable
  • Removing application variables 
    {application} class >> removing
	AppVariable1 shuttingDown.
	AppVariable1 := nil.

Design Pattern: Reading Serial Ports
  • Serial ports must be read constantly to keep from losing data.
  • Packets sent over serial port must be assembled, checked for errors.
  • Implement as two separate processes:
    Serial port reader
    High priority process that reads data from the serial port, sends it to the Packet assember actor.
    Packet assembler actor
    Lower priority process that reads the raw data from the Serial port reader, assembles into packets for higher level protocol.
  • Separation keeps things lively, but need to watch actor queue size ...

Voice Recognition Component
  • Wrapped in Smalltalk automagically from header files
    • 579 C #defined constants -- placed in a pool dictionary
    • 176 C functions -- wrapped in PlatformFunctions
    • 78 unique response 'objects' wrapped into Smalltalk classes
  • ViaVoices uses Windows events for asynchronous message (HWND/WM_COMMAND/etc)
  • Stole code from SocketCommunicationsInterface (SciPolledWinApp) to figure out how to create an object HWND in Smalltalk and handle messages to it.

Text-To-Speech Component
  • IBM ViaVoice includes Text-To-Speech (TTS) component called VirtualVoices.
  • VirtualVoices packaged as an ActiveX control.
  • Wrapped into Smalltalk with OleApplicationFramework.
  • Application variable created as single access point.
  • Existing Smalltalk classes such as Time, Date, Integer, Fraction extended with method speakString. speakString answers a string which can be 'spoken' by VirtualVoices. Examples:
    • Date today speakString -- 'Tuesday, April 14th 1998'
    • Time now speakString -- '11:05 p m'
    • (1/7) speakString -- '1 seventh'
  • TTS usage had to be coordinated with Voice Recognition and RealAudio since only one can use the sound card at a time.
  • Audio volume level temporarily reset to a 'standard' volume.

Port I/O Component
  • Needed the ability to write data on the parallel port.
  • Although Windows documentation would have you believe 'normal' i/o can be used to do this, it can't.
  • Only way is to use the C outp() function.
  • Not found in any existing DLL
  • So I wrote a 5 line C file, created a DLL from it, used it via PlatformFunction.

Java Communication Component
  • Needed a way to communicate with Java app running in the car and on the Web site.
  • TCP/IP sockets seemed to be the easiest transport.
  • Designed a simple string based protocol.
  • Socket open to send commands to peer.
  • Socket open to receive commands to peer.
  • Client sends command (string) to server, server sends response (string) back to client.
  • Commands designed to be asynchronous, so Actor pattern used to serial message sends to server.
  • Also had to handle retries.

PalmPilot Component
  • We cheated - we used existing PalmPilot HotSync application.
  • When requested, we read data from the PalmPilot databases.
  • PalmPilot database formats are not documented, so lots of guess work.
  • PalmPilot data was only spoken back to user, so classes like PalmPilotDate extended to generate appropriate string (1/1/98 becomes "December first 19 98".

Serial I/O Component
  • Used pre-wrapped Win32 classes under PlatformInterface.
  • Needed to use really ugly Win32 Overlapped I/O to enable simultaneous reading/writing.
  • Used ACO (EsAsynchronousCallOut) for one function to wait on i/o completion.
  • Also implemented Virtual Serial I/O ports
    • Modelled one end of a serial connection
    • Two ends connected by a set of queues
    • Allowed for testing multiple serial port interaction on machines with small number of ports (my Thinkpad has just one serial port!)

Instrument Panel Control Module Component
  • Read-only interface
  • Packets sent every 100ms.
  • Provided interface to:
    • Steering wheel buttons
    • Seat occupancy
    • Door ajar
  • Used the Reading Serial Ports Design Pattern

Mobile Media Link Component
  • Read-write interface
  • Packets sent as needed.
  • Provided interface to:
    • Amplifier
      • Volume / Balance / Fade
      • Audio source - Computer / CD Player / Phone
    • CD Player
      • Commands - Play / Stop / Next Track / etc
      • Track status - Current Disc / Current Track / Time Playing
    • Seat occupancy
    • Door ajar
  • Used Reading Serial Ports Design Pattern
  • Used Application Variable patterns for single access point to the MML
  • Used Actor pattern to asynchronously send commands

Class 2 Network Component
  • Class 2 network is GM's implementation of an SAE standardized in-car network.
  • Used to:
    • Lock/unlock/open doors
    • Dial the phone
    • Access car sensors
  • Used off-the-shelf protocol converter with serial interface.
  • Used Serial Port Reading pattern.

Head Up Display Component
  • Write-only interface
  • Actually controlled both the Head Up and Head Down displays (HUD and HDD)
  • Provided interface to all display updates on the Head Up and Head Down displays
    • road speed
    • engine speed
    • engine status
    • door lock/ajar
    • functions for the steering wheel buttons
    • indication of waiting e-mail
    • navigation information
    • microphone on/off
    • text message area for giving feedback to the driver
  • Used Application Variable patterns for single access point to the HUD
  • Used Actor pattern to asynchronously send commands

Configuration Component
  • Needed a way to configure system easily, from a packaged image.
  • Created a configuration component
  • Configuration read from a file (like a Windows .ini file).
  • Viewable, modifyable from packaged image.
  • Editor used EwTableList, existing edit policies to edit basic types (String, Boolean, Integer).
  • Configuration accessible via DelcoConfig application variable.
  • Ended up with 88 configurable settings.

Various Infrastructure Components
  • Message Logger
    • Log to window
    • Verbosity setting
    • Ceiling on number of message displayed on Window
    • Optional Log to disk
    • Used Application Variable pattern to provide single interface
    • Used Actor pattern to allow clients to issue requests to log messages, but have messages logged on lower priority process.
  • Process manager
    • Framework to build processes that blocked or delayed
    • Provided common API to start / stop / modify / enumerate processes
  • Control Panel
    • Shell to hold the Log
    • Menu items to start / stop / configure subsystems, testers, simulators, diagnostics

Using Simulators
  • Built simulators so the entire system could be tested on a Thinkpad at home
  • Less travel required to visit actual hardware
  • Simulators built for:
    • IPCM - provided a dialog to simulate seat occupancy, etc
    • MML - provided a text area to display incoming CD track status, 'remote control' to change amplifier settings and command the CD player
    • Class 2 - provided incoming 'garbage' data, as well as a way to trigger engine sensor failure
    • HUD/HDD - Textual simulation of information displayed on HUD and HDD

Packaging Tips
  • For Application Variable pattern, make sure variables are 'nilled' during packaging 
    MyApplication class >> packagingRulesFor: aCollector
   aCollector
      initializeToNilGlobalVariable: #MyApplicationVariable
  • Initialize class variables during packaging, if needed 
    MyApplication class >> packagingRulesFor: aCollector
   aCollector
      initializeToNilClassVariable: #MyClassVariable
      inClassNamed: #MyClass
  • Automatic version naming during packaging 
    MyApplication class >> packagingRulesFor: aCollector
	
   aCollector
      initializeClassVariable: 'VersionString'
      to: '''', self promptForVersionString, ''''
      inClassNamed: MyClass.
    Version string then accessed via the class variable in the packaged image.
  • Your own startup class
    • Subclassed the default EpRuntimeStartUp class, overrode #reportError:resumable:startBP:
      • to cause prompter on walkbacks to not appear
      • dump additional status information
    • Tracking garbage collection - see Process class>>addGCEventHandler:
